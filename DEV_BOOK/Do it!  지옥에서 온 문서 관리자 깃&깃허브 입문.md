02-2 버전 만들기

- 파일을 수정하고 저장할 때마다 이전 내용과 수정 내용을 구분하기 위하여 파일 이름을 달리 저장한다면, 관리가 복잡하고 번거롭다.
- 파일을 변경한 후 수정 내용과  버전을 만든 시간까지 기록할 수 있는 것이 깃과 같은 버전 관리 시스템이다.
- 깃에서 버전을 관리하면 원래 파일 이름은 그대로 유지하면서  파일에서 무엇을 변경했는지를 변경 시점마다 저장할 수 있다.
- 스테이지와 커밋 이해하기
    - 작업 트리 : 파일 수정, 저장 등의 작업을 하는 디렉터리.
    - 스테이지 : 버전으로 만들 파일이 대기하는 공간
    - 저장소 : 스테이지에서 대기하고 있던 파일을 버전으로 만들어 저장하는 곳
    - 스테이지와 저장소는 눈에 보이지 않으며 숨은 파일로 존재한다.
- 작업 트리에서 빔으로 문서 수정하기
    - $ git status : 깃의 현재 상태 확인
    - $ git add 파일이름.확장자 : 수정한 파일을 스테이징하기
    - $ git commit -m "메시지" : 메시지와 함께 스테이지에 있는 파일을 커밋하기
    - $ git commit -am  "메시지" : 스테이징과 커밋 과정을 한번에 처리함

02-3 커밋 내용 살펴보기 

```vim
$ git log : 커밋한 버전에 대한 설명 출력
$ git diff  : 스테이지에 있는 파일과 저장소에 있는 최신 커밋을 비교하여 차이점을 보여줌 - 커밋 전 최종 검토용
```

02-4 버전 만드는 단계마다 파일 상태 알아보기 

- tracked / untraced 파일 : 깃이 추적하고 있는/있지 않은 파일을 나타내며 한 번도 깃에서 버전 관리를 하지 않은 파일은 untracked 로 표시됨.

    참조 :  .gitignore 파일을 만들면 특정 확장자나 제목의 파일, 또는 디렉터리를 지정하여 버전 관리에서 제외시킬 수 있다. 

- unmodified, modified, staged 상태 :  각각 수정되지 않은 상태, 수정된 상태, 스테이지에 올라간 상태를 뜻함

    참조 : $ git commit —amend 로 직전에 커밋한 메시지를 수정할 수 있다. 

02-5 작업 되돌리기 
```vim
$ git checkout — 파일이름.확장자 : 작업 트리에서 수정한 파일을 git의 버전으로 되돌린다.
$ git reset HEAD 파일이름.확장자 : 수정된 파일을 스테이징 했을 때, 스테이징을 취소한다.
$ git reset HEAD^ : 깃과 스테이지를 최신 커밋을 하기 전 상태로 되돌린다.
    - soft HEAD^ : 최신 커밋을 하기 전 상태로 작업트리를 되돌린다.
    - mixed HEAD^ : 최근 커밋과 스테이징을 하기 전 상태로 작업 트리를 되돌린다. 옵션 없이 git reset  명령을 사용할 경우 이 옵션을 기본으로 적용한다.
    - hard HEAD^ : 최근 커밋과 스테이징, 파일 수정을 하기 전 상태로 작업 트리를 되돌린다. 이 옵션을 사용할 경우 되돌린 내용은 복구할 수 없다.
$git reset 커밋해시 : 특정 커밋 버전으로 되돌린다.
$git revert  커밋해시 : 커밋 삭제하지 않고 되돌린다.
```
03 깃과 브랜치 

03-1 브랜치란? 

- 고객사마다 다른 프로그램을 요구할 경우 고객사에 따라 제품과 사용설명서를 따로 관리한다면 중복자료가 많아지고 관리가 복잡해짐
- 이러한 단점을 없애고 버전을 여러 갈래로 나누어 관리할 수 있도록 하는 것이 브랜치 기능
- 분기 : 기존 내용에서 뻗어나가 만드는 새로운 파일
- 병합 : 분기했던 브랜치를 기존 내용 (MASTER)으로 합침

03-2  브랜치 만들기 
```vim
$ git branch : 깃에서 브랜치를 만들거나 확인하는 명령
  - 새 브랜치를 만들때는 git branch  브랜치명
$ git checkout 이동브랜치명 : 브랜치 사이를 이동하는 명령어
$ git log —oneline : 한 줄에 한 커밋씩 보여주어 로그를 간략하게 확인하고 싶을때 사용함
$ git log —oneline —branches : 각 브랜치의 커밋을 함께 보여줌
$ git log —oneline —branches —graph : 각 브랜치와 커밋의 관계를 그래프 형태로 보여줌
$ git log 1브랜치..2브랜치 || git log 2브랜치..1브랜치 : 브랜치 간의 차이점을 보여준다.
$ git merge 브랜치 : 현재 브랜치와 입력한 브랜치를 병합한다.
    - fast forward merge : 빨리 감기 병합. 분기 후 브랜치에 아무런 변화가 없는 경우 즉시 병합되는 것
    - $ git merge 브랜치명 —no —edit :편집기 창을 열지 않고 깃에서 지정하는 커밋 메시지를 그대로 사용함
    - $ git merge 브랜치명 —edit : 브랜치를 병합할 때 편집기 창이 나타나지 않도록 설정한 경우, 커밋 메시지를 추가하거나 수정하고 싶을 때 사용
    - 같은 파일의 같은 위치를 수정한 후 병합하는 경우 브랜치 충돌이 발생함
    - 이 경우 파일을 열어 충돌을 수정한 후 다시 커밋해야 함.
$ git branch -d 브랜치명 : 해당 브랜치를 삭제함
    - 특정 시점의 브랜치로 이동하고 싶은 경우, 해당 브랜치의 해시코드를 복사하여 git reset 해시코드라고 입력하면 이동할 수 있다.
$ git stash  : 커밋하지 않은 수정 내용을 임시 장소에 보관함.
    - 가장 먼저 감춘 것은 stash@{0}에 담긴다.
    - 다른 파일이 추가되면 기존 파일은 stash@{1} 로 옮겨지며, 새로 추가된 파일은 stash@{0} 에 담긴다.
    - stash stack 이라고도 불리는 이 임시장소는 가장 최근의 파일만을 꺼낼 수 있다.
    - 최근 파일을 꺼내려면 git stash pop 을 입력한다.
```

04-5 깃허브에 SSH 원격 접속하기 

- SSH : Secure Shell 의 줄임말로 보안이 강화된 안전한 방법으로 정보를 교환하는 방식
    - 공개키와 비밀키를 조합하여 강력한 보안 체계 구축
    - 깃허브에서 SSH 원격 접속이란 이러한 프라이빗 키와 퍼블릭 키를 사용해 현재 사용하고 있는 기기를 깃허브에 인증하는 것이다.
- SSH -원격 접속 구축 방법
    1. 홈 디렉토리 이동 
    2. ssh-keygen 입력 
    3. 파일 이름 입력하지 말고 엔터 키 누름 
    4. 두 번 더 엔터 키 누르면 프라이빗 키 경로와 퍼블릭 키 경로가 bash 에 출력된다. 
    5. 깃허브에 퍼블릭 키를 전송한다. 
        1. cd ~/.ssh 
        2. cat id_rsa.pub 
        3. 문자열을 복사한다. 
        4. 깃허브 - Settings - SSH and GPG keys - New SSH key 클릭 
        5. 퍼블릭 키를 붙여넣은 뒤 Add SHH key를 클릭하여 키를 추가한다. 
        6. 깃허브 비밀번호를 입력하여 등록을 마무리한다. 
- SSH 주소로 원격 저장소 연결하기
    1. 새 저장소를 생성한다. 
    2. 저장소가 만들어지면 HHTPS 주소가 나타난다. 옆의 SSH를 눌러서 SSH주소를 표시한 뒤 복사한다. 
    3. bash를 통해 SSH 방식으로 접속한다. 
        1. cd ~
        2. git init connect-ssh
        3. cd connect-ssh 
        4. git remote add origin 복사한 주소 붙여넣기

        참조 :  .gitignore 파일을 만들면 특정 확장자나 제목의 파일, 또는 디렉터리를 지정하여 버전 관리에서 제외시킬 수 있다. 

    - unmodified, modified, staged 상태 :  각각 수정되지 않은 상태, 수정된 상태, 스테이지에 올라간 상태를 뜻함

        참조 : $ git commit —amend 로 직전에 커밋한 메시지를 수정할 수 있다. 

    02-5 작업 되돌리기 

    - $ git checkout — 파일이름.확장자 : 작업 트리에서 수정한 파일을 git의 버전으로 되돌린다.
    - $ git reset HEAD 파일이름.확장자 : 수정된 파일을 스테이징 했을 때, 스테이징을 취소한다.
    - $ git reset HEAD^ : 깃과 스테이지를 최신 커밋을 하기 전 상태로 되돌린다.
        - —soft HEAD^ : 최신 커밋을 하기 전 상태로 작업트리를 되돌린다.
        - —mixed HEAD^ : 최근 커밋과 스테이징을 하기 전 상태로 작업 트리를 되돌린다. 옵션 없이 git reset  명령을 사용할 경우 이 옵션을 기본으로 적용한다.
        - —hard HEAD^ : 최근 커밋과 스테이징, 파일 수정을 하기 전 상태로 작업 트리를 되돌린다. 이 옵션을 사용할 경우 되돌린 내용은 복구할 수 없다.
    - $git reset 커밋해시 : 특정 커밋 버전으로 되돌린다.
    - $git revert  커밋해시 : 커밋 삭제하지 않고 되돌린다.

    03 깃과 브랜치 

    03-1 브랜치란? 

    - 고객사마다 다른 프로그램을 요구할 경우 고객사에 따라 제품과 사용설명서를 따로 관리한다면 중복자료가 많아지고 관리가 복잡해짐
    - 이러한 단점을 없애고 버전을 여러 갈래로 나누어 관리할 수 있도록 하는 것이 브랜치 기능
    - 분기 : 기존 내용에서 뻗어나가 만드는 새로운 파일
    - 병합 : 분기했던 브랜치를 기존 내용 (MASTER)으로 합침

    03-2  브랜치 만들기 

    - git branch : 깃에서 브랜치를 만들거나 확인하는 명령
        - 새 브랜치를 만들때는 git branch  브랜치명
    - git checkout 이동브랜치명 : 브랜치 사이를 이동하는 명령어
    - git log —oneline : 한 줄에 한 커밋씩 보여주어 로그를 간략하게 확인하고 싶을때 사용함
    - git log —oneline —branches : 각 브랜치의 커밋을 함께 보여줌
    - git log —oneline —branches —graph : 각 브랜치와 커밋의 관계를 그래프 형태로 보여줌
    - git log 1브랜치..2브랜치 || git log 2브랜치..1브랜치 : 브랜치 간의 차이점을 보여준다.
    - git merge 브랜치 : 현재 브랜치와 입력한 브랜치를 병합한다.
        - fast forward merge : 빨리 감기 병합. 분기 후 브랜치에 아무런 변화가 없는 경우 즉시 병합되는 것
        - git merge 브랜치명 —no —edit :편집기 창을 열지 않고 깃에서 지정하는 커밋 메시지를 그대로 사용함
        - git merge 브랜치명 —edit : 브랜치를 병합할 때 편집기 창이 나타나지 않도록 설정한 경우, 커밋 메시지를 추가하거나 수정하고 싶을 때 사용
        - 같은 파일의 같은 위치를 수정한 후 병합하는 경우 브랜치 충돌이 발생함
            - 이 경우 파일을 열어 충돌을 수정한 후 다시 커밋해야 함.
            - git branch -d 브랜치명 : 해당 브랜치를 삭제함
        - 특정 시점의 브랜치로 이동하고 싶은 경우, 해당 브랜치의 해시코드를 복사하여 git reset 해시코드라고 입력하면 이동할 수 있다.
        - git stash  : 커밋하지 않은 수정 내용을 임시 장소에 보관함.
            - 가장 먼저 감춘 것은 stash@{0}에 담긴다.
            - 다른 파일이 추가되면 기존 파일은 stash@{1} 로 옮겨지며, 새로 추가된 파일은 stash@{0} 에 담긴다.
            - stash stack 이라고도 불리는 이 임시장소는 가장 최근의 파일만을 꺼낼 수 있다.
            - 최근 파일을 꺼내려면 git stash pop 을 입력한다.

    04-1 원격 저장소와 깃허브 

    - 깃허브를 이용하면 원격 저장소에 파일을 저장하며 버전관리를 할 수 있다.
    - 깃허브 사용 방법

    04-2 깃허브 가입하기 

    1. [www.github.com](http://www.github.com) 에 접속한 후 sign up을 눌러 가입
    2. New repository 를 클릭하여 새 저장소 생성 
        - .gitignore : 특정 언어와 관련된 라이브러리나 실행 파일을 버전 관리시 무시하도록 설정해 주는 파일

    04-3 지역 저장소를 원격 저장소에 연결하기 

    1. 지역 저장소 만들기 
        - git init loc-git
        - cd loc-git
        - 임의의 파일 생성
        - git commit -m "add a"
        - git log
    2. 원격 저장소에 연결하기 
    - git remote add origin "https:// [github.com/](http://github.com/testloc)uploadrepository/test-1.git
    - git remote -v

    04-4 원격 저장소에 올리기 및 내려받기 

    - git push -u origin master
        - github의 해당 저장소에 가 보면 지역저장소의 파일이 업로드 되었음을 알 수 있다.
    - github 사이트에 직접 접속하여 파일을 커밋할 수도 있다.
        - Create new file - 파일 수정 - Commit new file
    - git pull origin master
        - 원격 저장소(origin) 에서 지역 저장소(master) 브랜치로 파일을 내려받는다.

    04-5 깃허브에 SSH 원격 접속하기 

    - SSH : Secure Shell 의 줄임말로 보안이 강화된 안전한 방법으로 정보를 교환하는 방식
        - 공개키와 비밀키를 조합하여 강력한 보안 체계 구축
        - 깃허브에서 SSH 원격 접속이란 이러한 프라이빗 키와 퍼블릭 키를 사용해 현재 사용하고 있는 기기를 깃허브에 인증하는 것이다.
    - SSH -원격 접속 구축 방법
        1. 홈 디렉토리 이동 
        2. ssh-keygen 입력 
        3. 파일 이름 입력하지 말고 엔터 키 누름 
        4. 두 번 더 엔터 키 누르면 프라이빗 키 경로와 퍼블릭 키 경로가 bash 에 출력된다. 
        5. 깃허브에 퍼블릭 키를 전송한다. 
            1. cd ~/.ssh 
            2. cat id_rsa.pub 
            3. 문자열을 복사한다. 
            4. 깃허브 - Settings - SSH and GPG keys - New SSH key 클릭 
            5. 퍼블릭 키를 붙여넣은 뒤 Add SHH key를 클릭하여 키를 추가한다. 
            6. 깃허브 비밀번호를 입력하여 등록을 마무리한다. 
    - SSH 주소로 원격 저장소 연결하기
        1. 새 저장소를 생성한다. 
        2. 저장소가 만들어지면 HHTPS 주소가 나타난다. 옆의 SSH를 눌러서 SSH주소를 표시한 뒤 복사한다. 
        3. bash를 통해 SSH 방식으로 접속한다. 
            1. cd ~
            2. git init connect-ssh
            3. cd connect-ssh 
            4. git remote add origin 복사한 주소 붙여넣기 

    05 깃허브로 협업하기 

    05-1 여러 컴퓨터에서 원격 저장소 함께 사용하기 

    - git hub 저장소에서 주소를 복사한 뒤 로컬 저장소에 같은 저장소를 만든다.
        - git clone *복사한 주소 붙여넣기*  폴더명
        - 여러 사람이 한 프로젝트를 같이 작업하고 있다면, 작업 전 원격 저장소에 새로 올라온 커밋을 가져와 지역 저장소의 버전이 원격 저장소와 일치하도록 한 뒤 작업을 시작한다. (git pull)
        - 작업을 마친 뒤 원격저장소에 커밋한다. (git commit -am "message" , git push

    05-2 원격 브랜치 정보 가져오기 

    - git fetch - 원격 저장소의 정보를 가져오는 기능이다.
    - full은 원격 저장소의 커밋을 가져와서 무조건 지역 저장소와 합친다면, 페치 명령은 원격 브랜치에 어떠한 변화가 있는지 그 정보만 가져온다.
    - 페치로 가져온 원격 저장소 정보는 FETCH_HEAD 라는 브랜치에 있다.
    - 페치 후 최신 커밋을 현재 브랜치에 합치려면 git pull, git merge 등의 명령을 사용할 수 있다.
    - git pull 은 git fetch 명령과 git merge FETCH_HEAD 명령 두 개를 합친 것과 같은 기능이다.

    05-3 협업의 기본 알아보기 

    - 깃허브 사이트에서 settings-collaborator 메뉴를 이용하여 공동 작업자들을 초대할 수 있다.
    - 공동 작업에서 사용할 이름과 이메일주소를 git [config-user.name](http://config-user.name) "사용자 이름" , git [config-user.email](http://config-user.email) 메일주소 로 정의한다.
    - 원활한 버전 관리를 위해 git log와 fetch 등등으로 저장소의 정보를 확인한 뒤 pull 혹은 push를 진행한다.

    05-4 협업에서 브랜치 사용하기 

    - 풀 리퀘스트를 통해 브랜치를 병합할 수 있다.
    - 브랜치의 상태와 누가 브랜치를 병합했는지 확인하려면 화면 위에 있는 branches를 클릭하여 알 수 있다.
    - 다른 작업자의 변경 내용을 바로 반영하기 위해 항상 pull 부터 한 뒤 자신의 작업을 진행하는 것이 좋다.

    06 깃허브에서 개발자와 소통하기 

    06-1 깃허브 프로필 관리하기 

    - 프로필 수정을 통해 기본 공개 정보를 수정할 수 있다.
    - "컨트리뷰션 그래프" 를 통해 타인에게 자신의 활동내역을 보여줄 수 있다.

    06-2 readme 파일 작성하기 

    - 사용자가 자신의 저장소를 보다 편리하게 살펴볼 수 있도록 안내문을 만들어 첫 화면에 띄워둔다.
    - 마크다운 문법을 사용해 단순히 텍스트만 나열할 수 있는 것이 아니라 여러 서식을 적용할 수 있다. (첨부 스크린샷 참조)
    - 이미지의 경우, 웹에 업로드 된 파일만 사용할 수 있다.
        - 로컬 저장소의 사진을 올리고 싶은 경우, 해당 저장소에 images 디렉터리를 만든 뒤 readme 에 ![아기고양이](./images/사진.jpg) 를 입력하여 사진을 업로드 할 수 있다.

    06-3 오픈 소스 프로젝트에 기여하기 

    - 깃허브에 공개된 오픈 소스 프로젝트에 번역, 소스 코드 수정등의 활동을 통해 기여할 수 있다.
    - 소스 코드 수정
        1. fork 기능을 통해 오픈 소스 저장소를 자신의 원격 저장소로 가져온다. 
        2. fork 한 저장소를 지역 저장소로 클론한다. 
        3. 소스코드를 수정한다. 
        4. 원본 오픈 소스 프로젝트로 pull request 한다. 
            - 커밋 메시지에 원본 저장소의 개발자에게 문서의 어떤 부분을 왜 수정했는지 설명한 내용을 입력하는 것이 좋다.

    06-4 깃허브에 개인 블로그 만들기 

    - 일반적인 웹 호스팅 서비스를 이용할 경우, 대부분 매달 일정 금액을 서버 회사에게 지불해야 한다.
    - 깃 허브에서는 GitHub Pages 라는 기능을 통해 저장소를 홈페이지로 사용할 수 있다.
    - 직접 홈페이지 파일을 사용하는 방법과, 깃허브에서 지원하는 Jekyll theme 를 사용하는 방법이 있다.
    - 원하는 jekyll theme 를 포크해온 뒤 저장소의 이름을 계정명/계정명.github.io 로 수정한다.
    - _config.yml 의 정보를 수정한다.
    - _posts 폴더 안에 새 글을 작성한다.
    - 사진을 넣고 싶은 경우, images 폴더에 원하는 파일을 넣은 뒤 포스트에 링크를 입력해 삽입한다.
